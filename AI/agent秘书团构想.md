# Agent 秘书团构想

> 总秘书大人统一调度，三官（问题官、知识官、复盘官）各司其职。数据全部落盘 Obsidian，可被 Dataview 查询。
>
> **文档维护规则**：每次秘书团功能更新后，同步更新本文档——新增/修改的功能、废弃的旧方案、Phase 状态变更都记录到位，保持文档与实际实现一致。

## 二、问题官（Question Officer）

**职责**：读取题源笔记，提炼知识点生成题目，写入题库。

### 2.1 触发方式

- 用户说"复盘"或"考试" → 总秘书调复盘官统计错题量 M → 问题官按 `新题量 = 8 - floor(M/5)`（最少 3）后台并行出题
- 用户说"出题" → 问题官独立手工出题

### 2.2 工作流程

1. 读取 `复习追踪/题源.md` 获取题源笔记列表
2. 识别 `#重点` 标签：重点笔记多出题 + 答对后不删除
3. 题型覆盖：选择、填空、简答、多选（每轮至少 1 道多选）
4. 题目写入 `复习追踪/题库.md`，每道题有唯一 ID
5. 同时写入 `复习追踪/问题状态库.md` 初始化状态记录

**抽题策略**：在题库中随机抽取，答过或答错过的知识点权重会调整。

**删除规则**：普通题答对即删；#重点 笔记的题答对不删。

### 2.3 题源.md 格式

```markdown
# 题源

- [[C语言基础]]
- [[C++]] #重点
- [[代码规范]]
```

`#重点` 标记 = 该笔记多出题 + 答对不删。

### 2.4 题库格式（含评分键）

每道题在 `复习追踪/题库.md` 中以独立条目存储。**评分键**在出题时同步生成，批改时直接匹配，选择题/填空题 90% 的情况无需调 LLM：

```markdown
### Q001 | 来源：[[C语言基础]] | 创建：2026-05-20

**类型**：选择题

**题目**：STM32的HAL_Delay()函数延时单位是什么？

**选项**：
A. 微秒  B. 毫秒  C. 秒  D. 纳秒

**答案**：B

**评分键**：
- 选 B → 正确
- 选 A → 错误（混淆了 HAL_Delay 和 delay_us）
- 选 C → 错误（数量级不对，1秒=1000ms）
- 选 D → 错误（混淆了延时单位和时钟周期）

---
### Q002 | 来源：[[C语言基础]] | 创建：2026-05-20

**类型**：填空题

**题目**：C语言中，`const int *p` 表示______。

**答案**：p 指向的值不可修改

**评分键**：
- 含"指向"/"指针"/"值"/"不可修改"/"只读"/"常量"中 2 个以上关键词 → 正确
- 答"p 本身不可修改" → 错误（混淆了 const int *p 和 int * const p）
- 答"p 是常量指针" → 错误（术语用反了，const int *p 是指向常量的指针）
```

**评分键规则**：
- 选择题：直接列出每个选项的判罚（正确/错误+错误原因）
- 填空题：列出关键词集合 + 常见错误答案及对应判罚
- 简答题：列出必须覆盖的要点，达到 80% 即合格（简答仍需 LLM 语义判断）

## 三、知识官（Knowledge Officer）

**职责**：以 Obsidian vault 为知识库回答用户问题；出题时为问题官提供知识点参考；批改时提供正确答案和解析。

### 3.1 两种工作模式

| 模式 | 触发 | 行为 |
|------|------|------|
| **问答模式** | 用户直接提问 | 搜索 vault，找到答案并回复，附带 `[[笔记链接]]` |
| **辅助批改** | 总秘书批改时遇到简答题或边界情况 | 提供标准答案 + 知识点解析 + 可选的工程小拓展 |

> 注意：选择题和填空题的批改由总秘书通过评分键直接完成，知识官只介入简答题和边界情况的语义判断。

### 3.2 实际批改流程（对话模式）

1. 总秘书逐题展示 → 用户回答
2. 选择题/填空题：总秘书直接用评分键匹配，即时输出对错 + 解析
3. 简答题/边界情况：总秘书调知识官做语义比对
4. 总秘书更新 `复习追踪/问题状态库.md`

### 3.3 关键约束

- 只基于 vault 中已有的知识回答，不编造
- 所有回答附带源笔记链接，方便溯源
- 批改时给出"为什么对/错"的解释

## 四、复盘官（Review Officer）

**职责**：管理 1-3-7 复习节奏——每天检查到期错题，统计错题量，与问题官协调新题量。

### 4.1 触发方式

- 用户说"复盘"或"考试" → 总秘书调复盘官
- 未来规划：Hook 每天定时自动触发

### 4.2 每日工作流（对话模式）

1. 复盘官读取 `复习追踪/问题状态库.md`，筛选 `下次复习 <= 今天` 的错题
2. 统计到期错题量 M，传递给问题官
3. 问题官计算新题量 = 8 - floor(M/5)（最少 3），后台并行出题
4. 复盘官开始逐题考错题，总秘书即时批改
5. 错题考完 → 新题已就绪 → 无缝衔接继续考
6. 答对的题 → 普通题删除，#重点题保留；答错的题 → 纳入 1-3-7 调度

**数量规则**：
- 到期错题：全部必做，不设上限
- 新题量 = 8 - floor(M/5)，最少 3 道

### 4.3 错题复习算法（1-3-7）

```
普通题：答对 → 直接删除（知识点量大，随机抽题自然会循环覆盖）
#重点题：答对 → 保留在题库，答错 → 纳入错题复习

错题复习节奏：
错题 → 隔天复习
   ↓ 正确
第3天复习
   ↓ 正确
第7天复习
   ↓ 正确 → 删除（#重点题回到题库保留）

规则：
- 任意一次错误 → 回到隔天复习
- 到期错题全部必做，不设日限
```

### 4.4 问题状态库

```markdown
| ID | 问题摘要 | 来源 | 创建日 | 状态 | 正确次数 | 下次复习 | 备注 |
|----|---------|------|--------|------|---------|---------|------|
| Q001 | HAL_Delay 单位 | [[STM32]] | 05-20 | 待复习 | 0 | 05-21 | — |
| Q002 | const int *p | [[C语言基础]] | 05-20 | 第3天 | 1 | 05-23 | — |
| Q003 | CAN arbitration | [[CAN总线]] | 05-18 | 第7天 | 2 | 05-25 | #重点 |
```

### 4.5 答题进度条

每道题展示时附带进度信息，让用户知道当前位置：

```
| 新题 3/8 | 多选 | C++ |
| 错题 2/4 | 填空 | 代码规范 |
```

格式：`| 新题/错题 当前/总数 | 题型 | 来源 |`

## 七、已决策事项

1. **删除策略**：普通题答对即删；#重点笔记的题答对保留。
2. **批改标准**：选择题选项匹配；填空题所有空都答对才算正确；简答题 80% 语义准确。
3. **答卷方式**：对话逐题考，即时判对错（非文件填写模式）。
4. **数量规则**：到期错题全部必做，不设上限。新题量 = 8 - floor(M/5)，最少 3 道。
5. **错题节奏**：1-3-7 循环，任意错误回退隔天。
6. **并行出题**：复盘官考错题时，问题官后台同步出新题。
7. **错题分级反馈**：完全答错建议回读笔记；部分正确正常解析。
8. **评分键**：出题时同步生成，选择题/填空题直接匹配，简答题走知识官语义判断。
9. **错选项必须解析**：选择、多选、判断等客观题中，每个错误选项/错误判断都要说明为什么错、正确的是什么。帮助用户不仅知道"选什么"，更知道"为什么其他选项不对"。

## 八、出题规范（已采纳）

1. **题干不用否定词**：避免"下列哪项不是…"设问；判断题同样不用"以下说法错误的是"等否定设问
2. **至少 1 道分析/评价层题目**：如指出代码缺陷、论证方案优劣
3. **出题后自检认知层覆盖**：记忆层 < 30%
4. **代码补全型填空题**：给上下文代码，留空关键语句
5. **选择题精简为 3 选项**：每个干扰项反映真实常见错误
6. **每轮至少 1 道多选**
7. **笔记中的代码示例可直接作为考题**
8. **小白友好讲解**：多用生活类比、拆步骤讲、少堆术语、讲"为什么"
9. **错选项必须解析**：选择、多选、判断等有错误选项的题型，批改时逐个解释每个错误选项/错误判断为什么错、正确的是什么——让用户从错误中学习
10. **待实现：判断题**：给出一个陈述，判断正误并说明理由。干扰陈述应反映真实常见误解，批改时同样解析"为什么错"
