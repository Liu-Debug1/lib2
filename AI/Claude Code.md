# Claude Code 使用指南

## 一、应用方式

1. **桌面端应用** — 独立桌面客户端，完整的 UI 体验
2. **网页端应用** — 通过浏览器访问 claude.ai/code
3. **IDE 插件** — VS Code / JetBrains 插件，内嵌在编辑器中
4. **终端（推荐）** — CLI 方式运行，更原生、功能更全

## 二、三种模式

| 模式       | 说明                    | 适用场景            |
| -------- | --------------------- | --------------- |
| **对话模式** | 默认模式，一问一答             | 日常编码、调试、探索      |
| **计划模式** | `/plan` — 先设计方案再编码    | 复杂功能、架构决策、多文件改动 |
| **循环模式** | `/loop <时间>` — 定时重复执行 | 监控部署、轮询状态、定期检查  |

> [!NOTE] 切换模式快捷键 `Shift + Tab`

## 四、效率技巧

| 技巧 | 说明 |
| --- | --- |
| 并行任务 | 同时让 Claude 做多个独立任务 |
| 批量编辑 | 一次描述多个改动，Claude 逐步处理 |
| 背景运行 | 长时间任务可在后台运行，不阻塞对话 |
| `! <命令>` | 直接执行 shell 命令并返回结果 |
| 绝对路径 | 使用绝对路径引用文件，避免歧义 |
| `> ` 引用 | 用 `> ` 引用之前的消息，Claude 优先关注 |

> 跳过所有权限弹窗：`claude  --dangerously-skip-permissions`（仅当前会话生效）

## 五、最佳实践

**避免的用法：**
- ❌ 模糊指令：「优化这段代码」→ 具体说明优化方向
- ❌ 过多无关上下文 → 只提供相关的文件和日志
- ❌ 一次要求太多改动 → 拆分成多个小任务

**推荐的用法：**
- ✅ 先讨论方案再实施（尤其复杂任务）
- ✅ 改动后主动审查 diff
- ✅ 测试覆盖关键路径
- ✅ 记录项目约定到 CLAUDE.md

## 六、模型配置（接入 DeepSeek）

| 配置项         | 填入模型                | 分配逻辑说明                                                               |
| ----------- | ------------------- | -------------------------------------------------------------------- |
| 主模型         | `deepseek-v4-flash` | 日常默认用速度快、成本低的 flash，适合代码补全、简单问答、API 查询等高频轻量任务                        |
| Haiku 默认模型  | `deepseek-v4-flash` | Haiku 本身是 Claude 的轻量化高速模型，完美匹配 flash 定位，做工具调用、文件读取等轻量操作              |
| Sonnet 默认模型 | `deepseek-v4-pro`   | Sonnet 是 Claude 中阶主力模型，对应 Pro 的中高等推理能力，处理常规代码编写、逻辑分析任务               |
| Opus 默认模型   | `deepseek-v4-pro`   | Opus 是 Claude 最高阶模型，对应 Pro 的最强推理能力，处理长上下文、代码重构、疑难 bug 排查、架构设计等高复杂度任务 |

## 七、会话管理

### 7.1 上下文压缩

建议上下文占比达到 **60%** 时主动压缩。

| 好处 | 说明 |
| --- | --- |
| **省钱** | Token 越少，API 费用越低，长对话场景降幅尤为明显 |
| **避免报错** | 不会因超限导致调用失败，保障对话连续性 |
| **提升质量** | 保留关键信息、去除冗余噪音，让模型聚焦核心内容，避免"遗忘早期决策" |
| **降低延迟** | 压缩后上下文更小，推理速度更快，用户体验更流畅 |
| **支持长任务** | 使 AI 能在单次会话中完成需要几十甚至上百轮交互的复杂工程 |

- `/compact` — 主动压缩对话历史
- `/context` — 查看上下文占比、token 占比

### 7.2 状态栏配置

在对话框输入以下内容，让 CC 配置 statusLine，可随时在底部看到上下文占比：

```
 帮我配一个 statusLine，能显示当前目录 + 模型 + 上下文剩余百分比
```

> [!NOTE] 配置后需重启 CC 生效

### 7.3 退出与恢复s

- **退出 CC**：连按两次 `Ctrl + C`
- **恢复历史对话**：
  - `/resume` — 从历史对话列表中选择恢复
  - `claude -c` — 直接回到上一次对话（continue）

### 7.4 用 Git + GitHub 保存项目状态

利用 Git 对项目进行版本控制，防止误删误改，也便于回溯和协作。

#### 7.4.1 初始化仓库

```
帮我在项目目录 git init，做一个初始 commit
```

或者更具体地说：

```
初始化 Git，忽略图片和 .obsidian 配置，然后做第一次提交
```

CC 会自动创建 `.gitignore` 并完成 `git init` + `git add -A` + `git commit`。

#### 7.4.2 日常提交

每次整理完笔记或改完代码后：`git commit`

或手动指定：`提交当前改动，commit 信息写"修复xxx"`


#### 7.4.3 推送到 GitHub

先关联远程仓库，然后推送：`推到 https://github.com/用户名/仓库名.git`
如果已有关联的远程，直接：`git push`

#### 7.4.4 常用操作速查

| 操作    | 对 CC 说            | 等价命令                       |
| :---- | :---------------- | :------------------------- |
| 查看状态  | `git status`      | `git status`               |
| 提交    | `git commit`      | `git add -A && git commit` |
| 看提交历史 | `git log`         | `git log --oneline`        |
| 恢复文件  | `用 git 恢复 xxx 文件` | `git checkout -- 文件`       |
| 看改了啥  | `git diff`        | `git diff`                 |
| 推到远程  | `git push`        | `git push`                 |

> [!tip] 建议
> 每次整理完笔记、写完重要代码后随手 commit + push，GitHub 就是你的免费远程备份。误删了也能 `git checkout` 恢复。

## 八、个性化配置

### 8.1 第一优先级：`CLAUDE.md` 文件

**全局级** — 所有项目通用的原则，位于 `C:\Users\Liuzwei\.claude`

> [!NOTE] 创建方式
> 1. 让 CC 自己写进全局 `CLAUDE.md`
> 2. 手动编辑 `C:\Users\Liuzwei\.claude\CLAUDE.md`

> [!NOTE] 内容建议
> - 不要塞入太多内容
> - 放最顶层、最稳定的长期执行内容
> - 逐步添加高频错误并修正

**项目级** — 单独的项目规则，位于项目的根目录下

> [!NOTE] 创建方式
> 1. 在工作目录下输入 `/init`，CC 会创建 `.claude` 文件夹和 `CLAUDE.md` 文件
> 2. 不用一开始就建立，可以等项目具备雏形之后再 `/init` 初始化

> [!NOTE] 内容建议
> - 跟随项目开发变化
> - 添加功能、更新要求
> - CC 踩过的坑同步更新到 CLAUDE.md

**子文件夹级**（可逐层递归）— 略

### 8.2 第二优先级：Auto-memory

`/memory` 手动打开 Agent 的 Auto-memory，主动记忆（记录在工作目录下的 `.claude` 文件中）。

	### 8.3 第三优先级：自行构建文档

自己构建文档，在对话中作为上下文引用即可。

## 十、注意事项

- Claude Code 使用 CommonMark 规范渲染 Markdown
- 对话有上下文窗口限制，过长时系统会自动压缩早期消息
- 文件编辑使用精确匹配工具，缩进和空格需要保持一致
- Windows 环境下使用 Unix shell 语法（正斜杠路径等）
