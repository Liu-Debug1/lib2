## 一、Git 和 GitHub 是什么关系

```
Git                      GitHub
┌──────────────┐        ┌──────────────────┐
│ 你电脑上的     │  推送   │ 同一个项目的        │
│ 版本管理工具   │ ────→  │ 云端备份 + 历史记录  │
│              │ ←────  │                    │
│              │  拉取   │                    │
└──────────────┘        └──────────────────┘
```

|        | Git            | GitHub          |
| ------ | -------------- | --------------- |
| 是什么    | 版本控制**工具**（软件） | 托管代码/笔记的**网站**  |
| 装在哪    | 你的电脑上          | 云端服务器           |
| 不联网能用吗 | 能              | 不能              |
| 作用     | 记录每次修改的历史      | 备份 + 多设备同步 + 协作 |

**一句话**：Git 负责记录历史，GitHub 负责云端备份。两者通过 `push`（上传）和 `pull`（下载）通信。

## 二、核心概念

### 2.1 仓库（Repository）

一个项目文件夹 + 它的所有历史记录，就是一个仓库。你的 `Ob_Learning` 就是一个仓库。

### 2.2 提交（Commit）

每做一次 `commit`，相当于给整个项目拍一张**快照**。以后随时可以回到这张快照的状态。

```
v1.0  →  v1.1  →  v1.2  →  v2.0
初稿     加笔记   改标题    大整理
```

### 2.3 分支（Branch）

分支就是**从主线上分叉出去的一条独立版本线**。类比写论文：

```
master（主线）：  初稿 → 二稿 → 三稿 → 终稿
                        ↘
分支（试验）：             换个写法试试 → 不错，合回去
                        ↘
                         改砸了 → 删掉分支，当没发生
```

**关键理解**：在分支上修改，不影响 `master`。合回去之前，两条线完全独立。

### 2.4 暂存区（Stage）

Git 的两步提交机制：

```
工作目录               暂存区                 历史记录
(你改的文件)  ──add──→ (准备提交的)  ──commit──→ (永久保存)
```

- `git add`：选哪些文件这次要提交
- `git commit`：正式记录到历史

#### 为什么不能一步提交？

Git 设计成两步，是为了让你**挑改动用哪个、不用哪个**。

假设你同时改了三样东西：

| 文件 | 状态 |
|------|------|
| C++笔记.md | 写完了，可以保存 |
| 论文第三章.md | 写了一半，不想保存 |
| 旧的草稿.md | 确定要删 |

如果只有一步，三者会混在一起提交。但有 `add` 就能挑：

```bash
git add "C++笔记.md"     # 只选写好的
git rm "旧的草稿.md"      # 确认删除
# 论文第三章.md 不 add，留在本地继续写
git commit -m "更新C++笔记，删旧草稿"
```

论文那一半的修改不动，下次继续写，干干净净。

> [!tip]
> `git add .` 是"全选"，适合只改了一个方向、想全提交的场景。
> 改了好几个不相干的东西时，分开 add、分开 commit，历史记录会更清晰。

## 三、常用操作

### 3.1 日常四步走（常用 90%）

```bash
git status                          # 看看改了什么
git add .                           # 把所有修改加入暂存区
git commit -m "写清楚这次改了什么"    # 提交到本地历史
git push                            # 上传到 GitHub 备份
```

### 3.2 分支操作

```bash
git branch              # 查看所有分支，当前分支有 * 标记
git checkout -b 分支名   # 创建新分支并切过去
git checkout master     # 切回 master
git merge 分支名         # 把分支的修改合进当前分支
git branch -d 分支名     # 删除已合并的分支
git branch -D 分支名     # 强制删除分支（放弃该分支的修改）
```

### 3.3 回退操作

```bash
git checkout -- 文件名       # 撤销某个文件的修改（回到上次 commit 的状态）
git reset --hard HEAD~1     # 撤销最近一次 commit（谨慎使用）
```

## 四、分支使用策略（个人项目）

### 什么时候不用分支

平常写笔记、改几个文件 —— **直接在 master 上干活**：

```
写笔记 → git add → git commit → git push
```

### 什么时候开分支

**大刀阔斧整理，怕改坏了**：

```bash
# 1. 从 master 分出一条试验分支
git checkout -b reorg-notes

# 2. 在里面随便改、随便删、随便试
# 3. 改完后切回 master
git checkout master

# 4. 效果好就合回去
git merge reorg-notes

# 5. 删掉用完的分支
git branch -d reorg-notes

# 6. 推送到 GitHub
git push
```

**搞砸了想放弃**：

```bash
git checkout master       # 切回 master
git branch -D reorg-notes # 删掉分支，一切恢复原样
```

### 你的原则

- **90% 的时间**：待在 `master`，直接写笔记，不用管分支
- **10% 的时间**：要做大规模重组时，开个分支当安全网
- 分支用完就删，保持清爽

## 五、常见问题

### 文件突然不见了？

很可能是切错了分支。`git branch` 看当前在哪个分支，切回正确的就行：

```bash
git checkout master
```

### 提示有未跟踪文件？

`.obsidian/` 和 `图片/` 被 `.gitignore` 排除，Git 不管它们，这是正常的。如果想跟踪新文件，用 `git add 文件名`。

### push 失败说分支没有上游？

第一次 push 新分支时需要关联远程：

```bash
git push -u origin master
```
之后直接用 `git push` 就行。

## 六、你的仓库结构

```
本地：
  master（主分支，所有笔记都在这里）

GitHub 远程：
  origin/master（和本地 master 同步）
  origin/main（旧分支空壳，可忽略）
```

正常只需在 `master` 上工作，不需要碰其他分支。
