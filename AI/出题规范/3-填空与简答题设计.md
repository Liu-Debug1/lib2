# 填空与简答题设计

> 来源：黄光扬《教育测量与评价》、柳博《考试命题制度研究》、Wozniak 20 Rules、NAEP 评估框架

## 一、填空题设计

### 核心原则

| 原则 | 说明 |
|------|------|
| 空白排布均匀 | 每道题最多 2 空，空白长度一致（防猜测） |
| 空出关键词 | 空白应是核心概念/术语，不是细枝末节 |
| 答案唯一 | 确保标准答案唯一，或列出可接受的所有等价表述 |
| 题干完整 | 填空后完整句子仍通顺、有意义 |

### 命题步骤

1. 写出一句完整的陈述句（含核心概念）
2. 挖掉关键词
3. 检查：挖掉后语境是否仍清晰指向唯一答案

### C/C++ 命题策略

**概念填空题**（挖术语）：
> C++ 中默认的成员访问权限：struct 为 ___，class 为 ___ 。

**代码填空题**（挖关键语法）：
> `new` 分配内存后需用 ___ 释放，`new[]` 配对的释放运算符是 ___。

**追踪填空题**（挖输出结果）：
> 以下代码的输出结果是 ___ 。
```cpp
int a = 5;
int &b = a;
b = 10;
cout << a;  // 输出：___
```

### 评分规则（我们已采用）

**所有空都答对才算正确**。任一空错 → 整题错。 → 这符合"一题一知识点"原则：多空的题应考的是同一个知识的多个维度，不是互不相关的拼凑。

## 二、简答题设计

### 核心原则

| 原则 | 说明 |
|------|------|
| 一题一知识点 | 不在一道简答题中考查多个不相关的知识点 |
| 问题明确、边界清晰 | 明确要求回答的范围和深度 |
| 标准答案按要点给分 | 最小评分点 1 分，列出所有可接受的等价表述 |
| 避免开放式 | 不用"谈谈你的理解"，改用"简述 X 的两个特点和原因" |

### 评分标准（我们已采用）

**80% 语义准确即合格**，不强求逐字匹配。

因为简答题的目的是**倒逼输出**，帮助用户检验是否真正理解，而非打分排名的标准化考试。

### C/C++ 命题策略

- 概念对比型：如"简述传值、传指针、传引用的区别"
- 原理解释型：如"为什么宏定义参数必须加括号？"
- 方案选择型：如"何时用 struct，何时用 class？给出判断依据"
- 代码分析型：如"这段代码有什么问题？如何修改？"

## 三、三种题型核心对比

| 维度 | 选择题 | 填空题 | 简答题 |
|------|--------|--------|--------|
| 考核层级 | 记忆/理解/应用 | 记忆/理解为主 | 分析/评价/创造 |
| 猜测概率 | 25%（四选一） | 低 | 无猜测余地 |
| 评分客观性 | 高 | 中 | 需标准答案参考 |
| 命题成本 | 高（设计干扰项） | 中 | 高（制定评分标准） |
| 适合场景 | 大面积知识点覆盖 | 核心概念精准检测 | 深度理解检验 |
