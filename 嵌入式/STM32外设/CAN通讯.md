# CAN 通信

CAN（Controller Area Network）是面向多控制器协同的串行总线。它最早由 Bosch 面向汽车电子提出，后来通过 ISO 11898、ISO 11519 等标准化，核心目标是让多个 ECU 在强干扰环境下用较少线束可靠通信。

一句话记忆：**CAN 不是“某个设备给某个设备发消息”，而是“节点把带 ID 的报文广播到总线，所有节点按 ID 选择是否接收”。**

## 一、CAN 解决什么问题

### 1.1 汽车为什么需要 CAN

传统点对点通信会让线束数量随 ECU 数量快速上升：
```text
点对点线束：
ECU1 ───── ECU2
ECU1 ───── ECU3
ECU2 ───── ECU3
ECU3 ───── ECU4

CAN 总线：
ECU1 ─┐
ECU2 ─┼── CAN_H / CAN_L ──┬─ ECU3
ECU4 ─┘                   └─ ECU5
```

CAN 的价值主要体现在：
- **减少线束**：典型 CAN 只需要 `CAN_H` 和 `CAN_L` 两根通信线。
- **支持多主通信**：总线空闲时，任意节点都能发起通信。
- **抗干扰能力强**：使用差分信号，适合汽车和工业环境。
- **实时性可控**：报文 ID 同时决定含义和优先级，高优先级报文可优先获得总线。
- **错误处理完善**：内置 CRC、ACK、位填充、错误帧、错误计数和 Bus-Off 隔离。
### 1.2 与 UART、I2C、SPI 的区别

| 协议   | 引脚               | 双工  | 时钟  | 电平  | 典型设备关系 | 典型应用               |
| :--- | :--------------- | :-- | :-- | :-- | :----- | :----------------- |
| UART | TX、RX            | 全双工 | 异步  | 单端  | 点对点    | 两个设备互相通信、调试串口      |
| I2C  | SCL、SDA          | 半双工 | 同步  | 单端  | 一主多从   | MCU 外挂低速传感器、EEPROM |
| SPI  | SCK、MOSI、MISO、SS | 全双工 | 同步  | 单端  | 一主多从   | 高速外设、屏幕、Flash      |
| CAN  | CAN_H、CAN_L      | 半双工 | 异步  | 差分  | 多主多节点  | 多个控制器互相通信          |

- 两个板子低速调试：优先 UART。
- 一个 MCU 挂多个低速芯片：优先 I2C。
- 一个 MCU 挂高速外设：优先 SPI。
- 多个控制器在噪声环境中互发状态和控制量：优先 CAN。

## 二、CAN 网络硬件层

### 2.1 节点结构

一个 CAN 节点通常由三层组成：

```text
应用程序
  │  读写报文 ID、DLC、Data
  ▼
CAN 控制器
  │  TX/RX 逻辑信号
  ▼
CAN 收发器
  │  差分电平
  ▼
CAN_H / CAN_L 总线
```

- **CAN 控制器**：负责帧格式、仲裁、CRC、ACK、错误处理、过滤器、发送邮箱、接收 FIFO 等协议细节。
- **CAN 收发器**：负责把控制器的 `TX/RX` 逻辑电平转换为 `CAN_H/CAN_L` 差分电平。
- **总线线缆**：通常用双绞线，长距离或强干扰环境可使用屏蔽双绞线。

STM32 的 bxCAN 属于 CAN 控制器，外部仍然需要 TJA1050、SN65HVD230、TJA1051 等 CAN 收发器。**STM32 的 CAN_TX/CAN_RX 不能直接接到 CAN_H/CAN_L。**

![[图片/CAN通讯/CAN收发器电路.png|717]]

### 2.2 高速 CAN 与低速 CAN

| 项目   | 高速 CAN                 | 低速 CAN             |
| :--- | :--------------------- | :----------------- |
| 典型速率 | 125 kbps ~ 1 Mbps      | 10 kbps ~ 125 kbps |
| 典型距离 | 1 Mbps 时约 40 m         | 40 kbps 时约 1 km    |
| 拓扑   | 闭环总线                   | 开环总线               |
| 终端电阻 | 两端各 `120Ω`             | 一端约 `2.2kΩ`        |
| 典型应用 | 动力、底盘、实时控制             | 车身舒适系统、低速控制        |
| 节点数量 | 受总线延迟和电气负载限制，常见标准中约几十个 | 通常更少               |

> [!warning] 终端电阻不是每个节点都加
> 高速 CAN 总线只在物理总线两端各接一个 `120Ω` 终端电阻。中间节点不应重复加终端，否则等效阻抗变小，会造成驱动负载过重和波形畸变。
### 2.3 差分电平与显性/隐性

CAN 总线用 `CAN_H - CAN_L` 的电压差表示逻辑位。

| 总线位 | 名称        | 逻辑值 | 总线特性           | 高速 CAN 典型差分电压 |
| :-- | :-------- | :-- | :------------- | :------------ |
| 显性位 | Dominant  | `0` | 有节点拉成 0，总线就是 0 | 约 `2V`        |
| 隐性位 | Recessive | `1` | 所有节点都释放，总线才是 1 | 约 `0V`        |
![[Pasted image 20260622175338.png|717]]
可以把 CAN 总线理解成“线与”：
```text
0 & X & X = 0
1 & 1 & 1 = 1
```

这就是 CAN 能实现非破坏性仲裁的物理基础：某节点发送隐性 `1`，但读回显性 `0`，说明有更高优先级节点正在发送，它就主动退出发送并转为接收。

### 2.4 硬件连接检查清单

调 CAN 前先检查硬件，很多“协议问题”其实是物理层问题：

- `CAN_TX`、`CAN_RX` 是否连接到收发器的 `TXD`、`RXD`。
- 收发器的 `CAN_H`、`CAN_L` 是否接反。
- 总线两端是否各有一个 `120Ω` 终端电阻。
- 断电测 `CAN_H` 与 `CAN_L` 之间等效电阻，高速 CAN 两端终端正常时约为 `60Ω`。
- 所有节点是否共地。CAN 是差分通信，但工程上通常仍需要参考地，避免共模电压超出收发器范围。
- 支线是否过长。高速 CAN 支线越短越好，避免反射。
- 收发器供电电压和 MCU IO 电平是否匹配。

## 三、CAN 协议基本概念

### 3.1 CAN 在 OSI 模型中的位置

CAN 主要覆盖：

- **物理层**：位时序、同步、位编码、差分信号、收发器、电缆等。
- **数据链路层**：帧格式、仲裁、ACK、CRC、错误检测、错误通知、错误恢复等。
- **部分传输控制思想**：硬件自动重传、错误恢复。

CAN 本身不规定“哪个 ID 表示车速、哪个字节表示方向盘角度”。这些属于应用层协议，常见做法是用 DBC、CANopen、J1939、UDS 等进一步定义。

### 3.2 报文 ID 不是节点地址

CAN 报文 ID 的核心含义：

- 表示**消息类型**，例如车速、轮速、制动压力、转角传感器状态。
- 决定**仲裁优先级**，ID 数值越小，优先级越高。
- 供接收节点的**过滤器**判断是否接收。

不要把 ID 设计成“发送方地址”或“接收方地址”。更工程化的设计是：

```text
ID = 信号功能 + 优先级 + 周期属性
Data = 具体数据字段
```

### 3.3 标准帧与扩展帧

| 类型  | ID 长度  | 适用场景              | 备注                    |
| :-- | :----- | :---------------- | :-------------------- |
| 标准帧 | 11 bit | 车身、底盘、实验小网络       | ID 范围 `0x000 ~ 0x7FF` |
| 扩展帧 | 29 bit | J1939、复杂网络、更多消息类型 | ID 范围更大，但仲裁段更长        |

在标准帧 ID 和扩展帧高 11 位相同的情况下，**标准帧优先级高于扩展帧**。这是通过扩展帧中的 `SRR`、`IDE` 等位保证的。

## 四、CAN 帧格式

### 4.1 五种帧类型

| 帧类型 | 用途              | 工程理解                   |
| :-- | :-------------- | :--------------------- |
| 数据帧 | 发送节点主动广播数据      | 最常用，周期报文和事件报文基本都靠它     |
| 遥控帧 | 接收节点请求某个 ID 的数据 | 传统 CAN 支持，但现代工程中使用频率不高 |
| 错误帧 | 节点检测到错误时通知全网    | 用来破坏当前错误帧并触发重发         |
| 过载帧 | 接收节点通知自己暂时没准备好  | 用于延迟后续发送，实际应用较少关注      |
| 帧间隔 | 分隔数据帧/遥控帧       | 至少 3 个隐性位              |

### 4.2 标准数据帧结构

![[图片/CAN通讯/CAN数据帧结构.png|732]]

标准数据帧可理解为：

```text
标准帧：
SOF | 11-bit ID | RTR | IDE | r0 | DLC | Data | CRC | ACK | EOF

扩展帧：
SOF | 11-bit ID | SRR | IDE | 18-bit ID | RTR | r1 | r0 | DLC | Data | CRC | ACK | EOF
```

| 字段     | 位数       | 作用                        |
| :----- | :------- | :------------------------ |
| SOF    | 1        | 帧起始，显性位，打破平静              |
| ID     | 11       | 标识符，决定消息含义和优先级            |
| RTR    | 1        | 数据帧为显性 `0`，遥控帧为隐性 `1`     |
| IDE    | 1        | 标准帧为显性 `0`，扩展帧为隐性 `1`     |
| r0/r 1 | 1 或 2    | 保留位，经典 CAN 中通常发送显性 `0`    |
| DLC    | 4        | 数据长度码，传统 CAN 数据长度为 0~8 字节 |
| Data   | 0~64 bit | 数据段，传统 CAN 为 0~8 字节       |
| CRC    | 15 + 1   | CRC 校验序列和 CRC 界定符         |
| ACK    | 1 + 1    | ACK 槽和 ACK 界定符            |
| EOF    | 7        | 帧结束，7 个隐性位                |

> [!note] DLC 注意点
> 传统 CAN 的有效数据最多 8 字节。DLC 为 9~15 时，传统 CAN 接收端通常不把它当成格式错误，但工程上不要这样设计。CAN FD 才支持更长数据段。

#### 4.2.1 控制场要重点记的位

标准数据帧的控制场是 `IDE + r0 + DLC`，总共 6 bit：

```text
IDE | r0 | DLC3 DLC2 DLC1 DLC0
 0  | 0  | 数据长度码
```

- `IDE = 0`：表示标准格式，使用 11 bit ID。
- `r0 = 0`：保留位，经典 CAN 中发送方通常必须发显性 `0`；应用层不要拿它传信息。
- `DLC`：只表示数据段字节数，不表示数据内容含义。
- `DLC = 0` 时，数据帧可以没有数据段，此时帧本身仍然是数据帧，和遥控帧靠 `RTR` 区分。

> [!warning] 保留位不要乱用
> `r0/r1` 的核心意义是为协议升级预留空间。经典 CAN 中发送端一般发显性 `0`，但接收端为了兼容性可能不把保留位的隐性/显性组合直接当作错误。做通信协议设计时，不要把保留位设计成业务信号。

#### 4.2.2 标准帧与扩展帧的关键区别

标准帧和扩展帧最大的区别不是数据段长度，而是 **ID 长度和仲裁过程不同**：

| 对比项        | 标准数据帧              | 扩展数据帧                  |
| :---------- | :----------------- | :--------------------- |
| ID 长度       | 11 bit             | 29 bit，即 11 bit 基本 ID + 18 bit 扩展 ID |
| 格式标志       | `IDE = 0`          | `IDE = 1`              |
| 保留位        | `r0`               | `r1`、`r0`              |
| 同高 11 位竞争时 | 标准帧优先              | 扩展帧因 `SRR/IDE` 为隐性位而让出总线 |
| 数据长度       | 传统 CAN 同样是 0~8 字节 | 传统 CAN 同样是 0~8 字节     |

工程速记：

- **标准帧更短，仲裁和传输开销更小**，车身、底盘、实验小网络优先考虑标准帧。
- **扩展帧 ID 空间更大**，适合 J1939 这类需要编码更多网络信息的协议。
- 当标准帧 ID 和扩展帧基本 ID 相同，标准帧会赢得仲裁，所以混用标准帧和扩展帧时要提前规划 ID。

#### 4.2.3 解析波形时的顺序

拿到 CAN 分析仪或示波器波形时，可以按这个顺序拆：

1. 找到 `SOF`：总线空闲后第一个显性位。
2. 读 `11-bit ID`：从高位到低位发送，得到标准帧 ID。
3. 看 `RTR`：`0` 是数据帧，`1` 是遥控帧。
4. 看 `IDE`：`0` 是标准帧；如果是 `1`，后面按扩展帧继续解析。
5. 看 `DLC`：决定后面读几个数据字节。
6. 读 `Data`：按字节整理业务数据，再结合协议矩阵解释信号。
7. 检查 `CRC/ACK/EOF`：确认这一帧是否正常结束，以及是否有节点返回 ACK。

![[图片/CAN通讯/CAN标准数据帧波形实例.png|732]]

### 4.3 ACK 不是软件回复 

ACK 段由发送节点先发送隐性位，接收正确的节点在 ACK 槽拉一个显性位。

关键点：

- 发送节点自己不 ACK 自己。
- 只要至少一个非休眠、非 Bus-Off 的接收节点正确收到，就会返回 ACK。
- 如果总线上只有一个节点在发，通常会出现 ACK 错误，因为没人帮它拉 ACK。
- ACK 只能说明“有节点正确收到帧”，不说明“应用层已经处理了数据”。

因此调试单节点 CAN 发送时，看到 ACK Error 不一定是代码错，可能只是总线上没有第二个正常节点。

### 4.4 遥控帧

遥控帧用于请求同 ID 的数据帧：

- 遥控帧没有数据段。
- `RTR = 1`，数据帧 `RTR = 0`。
- 遥控帧的 DLC 表示期望对方返回的数据长度。
- 当同 ID 的数据帧和遥控帧同时竞争时，数据帧优先级更高，因为数据帧的 RTR 是显性位。

现代工程里更常见的是“周期发送 + 事件触发发送”，遥控帧使用较少。做协议设计时，不要默认依赖遥控帧，除非网络和节点都明确支持。

## 五、仲裁机制

### 5.1 先占先得

总线**空闲**时，**任意**节点都可开始发送。节点检测到连续隐性位并满足帧间隔要求后，认为总线空闲。   

如果一个节点已经开始发送数据帧或遥控帧，其他节点不能再插入数据帧或遥控帧，只能等待下一次总线空闲。错误帧和过载帧是例外，它们可以破坏当前传输。

> [!note] 如何判断总线空闲？
>如果总线有连续11个隐性电平，则说总线空闲

### 5.2 非破坏性仲裁

多个设备（节点）*同时发送*时或*因等待*而同时发送，从仲裁段(ID号)第一位开始逐位比较：
- 规则：
	CAN总线协议会根据ID号（仲裁段）进行非破坏性仲裁，**ID号小的（优先级高）** 取到总线控制权，**ID号大的（优先级低）** 仲裁失利后将转入*接收状态*，等待下一次总线空闲时再尝试发送
- 发送显性 `0` 并读回 `0`：继续发送。
- 发送隐性 `1` 但读回 `0`：仲裁失败，立即停止发送，转入接收。
- 发送隐性 `1` 并读回 `1`：暂时继续。

🏅**实现非破坏性仲裁需要两个要求：**
1. 线与特性：总线任何一个设备发送显性电平0时，总线就会呈现显性电平0状态，只有当所有的设备都发送隐性电平1时，总线才会呈现隐性电平1状态     
2. 回读机制：每一个设备发送一个数据位后，都会读回总线当前的电平状态，以确认自己发出的电平是否被真实地发送出去了，根据线与特性，发出0回读必然是0，发出1不一定回读到1


*示例：*
```text
节点 A ID = 0x120 = 001 0010 0000
节点 B ID = 0x100 = 001 0000 0000

逐位比较到差异位：
节点 A 发送 1
节点 B 发送 0
总线实际为 0

结果：节点 A 发现自己发 1 读回 0，仲裁失败；节点 B 继续发送。
```

这叫非**破坏性仲裁**，因为获胜节点的报文没有被破坏，不需要重新开始。

![[图片/CAN通讯/CAN数据帧与遥控帧仲裁.png|711]]

> [!caution] 数据帧和遥控帧的优先级
> 数据帧和遥控帧**ID号**一样时，数据帧的优先级高于遥控帧

### 5.3 ID 规划原则

ID 规划不是随便编号，而是系统设计的一部分：

- 安全关键、周期短、时延敏感的报文使用更小 ID。
- 低频诊断、标定、日志类报文使用更大 ID。
- 同一功能域保持连续 ID 段，方便过滤和维护。
- 不要让高频低价值报文占用过高优先级，否则会挤压关键控制报文。

线控底盘示例：

| 报文类型           | 周期       | 优先级建议 | 原因           |
| :------------- | :------- | :---- | :----------- |
| 制动/转向控制命令      | 5~10 ms  | 很高    | 直接影响车辆稳定性和安全 |
| 轮速、横摆角速度、方向盘角度 | 5~10 ms  | 很高    | 控制算法核心反馈量    |
| 执行器状态、故障状态     | 10~20 ms | 高     | 用于闭环保护和故障降级  |
| 标定参数、诊断请求      | 事件触发     | 低     | 不应阻塞实时控制     |

## 六、位时序

### 6.1 位填充

CAN 使用 NRZ 编码，如果连续很长时间没有跳变，接收端容易失去采样时机。为此 CAN 规定：

- 在数据帧和遥控帧的 `SOF ~ CRC Sequence` 范围内，连续 5 个相同电平后，发送方自动插入 1 个反向填充位。
- 接收方检测到填充位后自动删除，还原原始数据。
- 如果应插入填充位的位置仍然出现第 6 个相同电平，则检测为填充错误。

示意：

```text
原始数据： 100000110
实际发送： 100000 1 110
                ↑
              填充位
```

位填充的作用：

- 给接收端提供足够边沿，方便再同步。
- 防止长时间电平不变导致采样点漂移。
- 让错误帧、过载帧这种“连续显性位”更容易被识别为异常。

### 6.2 位时序

CAN 没有独立时钟线，所有节点必须约定同一波特率。对每一位的数据时长进行了分段，同步段（SS）、传播时间段（PST）、相位缓冲段1（PBS1）和相位缓冲段2（PBS2),每个段由于若干个最小时间单位（Tq）构成
```text
1 bit = SS + PTS + PBS1 + PBS2

SS   : Synchronization Segment，同步段，固定 1Tq
PTS  : Propagation Time Segment，传播时间段，吸收总线传播延迟
PBS1 : Phase Buffer Segment 1，相位缓冲段 1
PBS2 : Phase Buffer Segment 2，相位缓冲段 2
SJW  : Resynchronization Jump Width，再同步补偿宽度
```

`SJW` 可以理解为**接收节点单次再同步时允许调整本地位时序的最大步长**：

- 单位是 `Tq`，常见配置如 `1Tq`、`2Tq`、`3Tq`、`4Tq`。
- 它不是一个独立的数据位时间段，不会直接增加正常 bit 的长度。
- 只有发生再同步时才起作用，用来限制这一次最多能把采样点往前或往后修正多少。
- `SJW` 太小：时钟误差或传播延迟稍大时，接收方可能来不及修正采样点。
- `SJW` 太大：同步调整过猛，会压缩相位缓冲段的时序裕量，反而可能引入不稳定。

采样点位于 `PBS1` 结束处：
![[图片/CAN通讯/CAN位时序与采样点.png|711]]

常用经验：

- 采样点常放在 75%~87.5% 附近，实际以芯片手册和总线长度为准。
- 总线越长，传播延迟越明显，`PTS + PBS1` 通常需要更长。
- `SJW` 通常结合晶振误差、线缆长度、节点数量和芯片手册推荐值配置。

### 6.3 硬同步与再同步

CAN 接收端通过总线边沿修正本地采样时刻：

- **硬同步**：
	- 总线空闲后检测到 `SOF` 的隐性到显性边沿，所有接收节点都会用这个边沿校准自己的本地位时序计数器，相当于把“秒表”重新对齐到 `SS` 段起点。硬同步只在帧起始处发生。

-  **再同步**：
	- 帧传输过程中，接收节点检测到有效边沿后，会修正自己的本地位时序；通过加长 `PBS1` 或缩短 `PBS2` 把采样点拉回正确位置，单次调整量不超过 `SJW`。

![[图片/CAN通讯/CAN硬同步.png|711]]


### 6.4波特率计算

$$
\text{波特率} = \frac{1}{T_{SS} + T_{PTS} + T_{PBS1} + T_{PBS2}} = \frac{1}{一个数据位时长}
$$

- $T_{SS}$：同步段时间
- $T_{PTS}$：传播时间段
- $T_{PBS1}$：相位缓冲段 1 时间
- $T_{PBS2}$：相位缓冲段 2 时间

例如，若 `Tq = 0.5us`，各时间段配置如下：

| 时间段 | Tq 数 | 换算时间 | 作用 |
| :--- | :--- | :--- | :--- |
| $T_{SS}$ | `1Tq` | `0.5us` | 同步段，硬同步时对齐位起点 |
| $T_{PTS}$ | `3Tq` | `1.5us` | 传播时间段，补偿总线传播延迟 |
| $T_{PBS1}$ | `3Tq` | `1.5us` | 相位缓冲段 1，采样点前的调整余量 |
| $T_{PBS2}$ | `3Tq` | `1.5us` | 相位缓冲段 2，采样点后的调整余量 |
| 合计 | `10Tq` | `5us` | 一个完整数据位时间 |

因此：

$$
\text{波特率} = \frac{1}{0.5us + 1.5us + 1.5us + 1.5us} = 200kbps
$$

## 七、错误处理机制

### 7.1 五类错误

| 错误类型   | 检测条件                  | 常见原因                       |
| :----- | :-------------------- | :------------------------- |
| 位错误    | 发送电平和读回总线电平不一致        | 总线冲突、物理层异常、收发器问题           |
| 填充错误   | 应位填充范围内出现连续 6 个相同电平   | 波形畸变、采样点错误、干扰              |
| CRC 错误 | 接收端计算 CRC 与帧中 CRC 不一致 | 干扰、位时序不匹配、线缆质量差            |
| 格式错误   | 固定格式位不符合规定            | 波特率错误、错误帧干扰、控制器异常          |
| ACK 错误 | 发送端在 ACK 槽读到隐性位       | 无接收节点、过滤不影响 ACK 但节点异常、接线错误 |

注意两个例外：

- 仲裁段中发送隐性位但读到显性位，是仲裁失败，不是位错误。
- ACK 槽中发送端发送隐性位但读到显性位，是收到 ACK，不是位错误。

### 7.2 错误帧

错误帧由错误标志和错误界定符组成：

| 状态     | 错误标志   | 含义                  |
| :----- | :----- | :------------------ |
| 主动错误状态 | 6 个显性位 | 积极破坏当前错误帧，让所有节点知道出错 |
| 被动错误状态 | 6 个隐性位 | 节点仍可通信，但不再强烈干扰其他节点  |

错误界定符为 8 个隐性位。

检测到错误后，发送节点通常会自动重发报文。对应用程序来说，这意味着：

- 发送函数返回成功，不一定代表物理总线已经最终成功发送，需结合邮箱状态和错误状态判断。
- 自动重传可能提升可靠性，但在故障总线上也可能造成总线占用。

### 7.3 错误状态与 Bus-Off

每个节点内部维护：

- `TEC`：发送错误计数器。
- `REC`：接收错误计数器。

| 节点状态    | 计数条件                        | 通信能力              |
| :------ | :-------------------------- | :---------------- |
| 主动错误状态  | `TEC <= 127` 且 `REC <= 127` | 正常通信，检测错误时发主动错误标志 |
| 被动错误状态  | `TEC >= 128` 或 `REC >= 128` | 可通信，但错误通知能力受限     |
| Bus-Off | `TEC >= 256`                | 禁止收发，节点从总线上隔离     |

![[图片/CAN通讯/CAN错误状态转换.png|550]]

Bus-Off 是 CAN 的故障封闭机制。它的意义是：当某个节点持续出错时，系统宁愿把它踢出总线，也不允许它一直破坏整网通信。

### 7.4 排故优先级

1. **硬件层**：`CAN_H/CAN_L` 是否接反，终端电阻是否正确，收发器是否供电，共地是否可靠。
2. **波特率层**：所有节点波特率和采样点是否一致，APB 时钟是否算错。
3. **模式层**：是否还在初始化模式、睡眠模式、静默模式、环回模式。
4. **ACK 层**：总线上是否至少有一个正常接收节点。
5. **过滤器层**：能发送但收不到时，再检查过滤器、FIFO、中断。
6. **错误状态层**：读取 `TEC/REC/Bus-Off`，判断是发送侧问题还是接收侧问题。
7. **负载层**：检查报文周期、ID 优先级和总线利用率。

## 八、STM32 bxCAN 外设

### 8.1 bxCAN 能做什么

STM32 常见 bxCAN 外设支持 CAN 2.0A/2.0B。以 STM32F103 系列为例，典型资源包括：

- 最高 1 Mbps。
- 3 个发送邮箱。
- 2 个接收 FIFO，每个 FIFO 3 级深度。
- 14 个过滤器组，互联型部分型号可到 28 个。
- 支持标准帧和扩展帧。
- 支持自动重传、自动离线恢复、自动唤醒、时间触发通信。
- 支持正常、静默、环回、环回静默等测试模式。

bxCAN 的基本结构：![[Pasted image 20260705135535.png]]


```text
CPU 写入
  │
  ▼
发送邮箱 0/1/2 ──► 发送控制器 ──► CAN_TX ──► 收发器

收发器 ──► CAN_RX ──► 接收控制器 ──► 过滤器 ──► FIFO0/FIFO1 ──► CPU 读取
```

### 8.2 发送流程

发送的基本流程：
1. 找到空闲发送邮箱。
2. 写入 ID、帧类型、DLC、Data。
3. 请求发送。
4. 硬件自动完成仲裁、位填充、CRC、ACK 检查和错误处理。
5. 软件检查发送完成、中止、错误或邮箱空中断。

### 8.3 接收流程

接收的基本流程：
1. 总线上出现报文。
2. CAN 控制器完成物理层和数据链路层检查。
3. 报文匹配过滤器。
4. 进入 FIFO0 或 FIFO1。
5. CPU 轮询或中断读取。

工程注意：
- FIFO 只有 3 级深度，中断处理慢会溢出。
- 高频报文要优先用过滤器分流，避免 CPU 被无关报文打爆。
- 接收回调中少做耗时计算，建议只搬运报文到软件队列，业务处理放到主循环或任务线程。

### 8.4 关键配置位

| 配置项 | 含义 | 工程建议 |
|:---|:---|:---|
| `NART` | 禁止自动重传。置 1 时只发送一次；置 0 时失败会自动重传 | 普通可靠通信常开自动重传；测试故障或实时系统需评估 |
| `TXFP` | 发送优先级。置 1 按请求顺序；置 0 按 ID 优先级 | 控制系统建议按 ID 优先级 |
| `RFLM` | FIFO 锁定。置 1 溢出丢弃新报文；置 0 溢出覆盖最后报文 | 状态量可覆盖，事件/故障报文更应避免丢失 |
| `ABOM` | Bus-Off 自动恢复 | 调试可打开；安全系统应结合故障策略决定 |
| `AWUM` | 自动唤醒 | 低功耗场景有用 |
| `TTCM` | 时间触发通信 | 需要时间戳或 TTCAN 思路时使用 |
### 8.5 工作模式

| 模式     | 用途             | 典型使用         |
| :----- | :------------- | :----------- |
| 初始化模式  | 配置 CAN 外设，禁止收发 | 设置波特率、过滤器、模式 |
| 正常模式   | 正常参与总线通信       | 实际运行         |
| 睡眠模式   | 低功耗，CAN 时钟停止   | 低功耗节点        |
| 静默模式   | 只监听，不影响总线      | 总线分析、抓包      |
| 环回模式   | 自发自收           | 不接外部总线的自测试   |
| 环回静默模式 | 内部自测且不影响外部总线   | 热自检          |

调试顺序：
1. 先用环回模式验证控制器初始化和收发软件逻辑。
2. 再用正常模式连接两块板或 CAN 分析仪。
3. 需要监听未知总线时用静默模式，避免误发破坏现场网络。

### 8.6 波特率计算

注意：
- STM32中，将PTS段和PBS1段合并为 $t_{BS1}$ 段

![[Pasted image 20260703150814.png]]


STM32F1 bxCAN 常见公式：
```text
BitRate = APB1_CLK / (BRP + 1) / (1 + (TS1 + 1) + (TS2 + 1))
		= 36MHz / 预分频系数 / （1+ tbs1 + tbs2）
```

### 8.7 过滤器

过滤器决定哪些报文进入接收 FIFO。它不影响 ACK：即使软件过滤掉某 ID，只要控制器正确收到帧，仍可能参与 ACK。

过滤器常见模式：

| 模式 | 含义 | 适合场景 |
|:---|:---|:---|
| 列表模式 | 明确列出要接收的 ID | 少量离散 ID |
| 屏蔽模式 | ID 与 Mask 匹配 | 连续 ID 段、功能域过滤 |
| 16 位宽 | 一个过滤器组可放更多标准 ID | 标准帧为主，ID 数较多 |
| 32 位宽 | 完整表达标准/扩展帧和控制位 | 扩展帧、精确匹配 |

屏蔽模式理解：

```text
(接收 ID & Mask) == (过滤 ID & Mask)
```

Mask 中：

- `1` 表示该位必须匹配。
- `0` 表示该位不关心。

设计建议：

- 实验阶段可先全接收，确认总线有数据。
- 工程阶段不要长期全接收，尤其是高负载网络。
- 安全关键报文建议单独过滤到固定 FIFO，并监控周期超时。

## 九、工程设计与调试

### 9.1 CAN 报文设计模板

设计一个 CAN 应用层协议时，至少定义：

| 项目          | 说明                            |
| :---------- | :---------------------------- |
| ID          | 标准/扩展、优先级、功能含义                |
| 周期          | 周期发送、事件发送或请求响应                |
| DLC         | 数据长度                          |
| Byte/Bit 定义 | 每个信号的起始位、长度、大小端、缩放因子、偏移量      |
| 有效范围        | 最大值、最小值、非法值                   |
| 超时策略        | 多久没收到视为失效                     |
| 校验策略        | 是否增加 rolling counter、checksum |
| 故障降级        | 数据失效后控制器如何处理                  |

示例：

| ID | 周期 | 信号 | 数据定义 | 超时 |
|:---|:---|:---|:---|:---|
| `0x101` | 10 ms | 四轮轮速 | `uint16_t`，0.01 km/h/bit | 30 ms |
| `0x120` | 10 ms | 横摆角速度、纵向加速度 | 有符号定点数 | 30 ms |
| `0x201` | 5 ms | 线控制动压力命令 | 目标压力 + rolling counter + checksum | 15 ms |
| `0x301` | 100 ms | 节点故障状态 | bit flag | 300 ms |


### 9.2 常见故障现象

| 现象 | 优先怀疑 | 检查方法 |
|:---|:---|:---|
| 发送一直失败，ACK Error | 总线上没有正常接收节点 | 接第二节点或 CAN 分析仪；确认正常模式 |
| 完全收不到 | 接线、波特率、过滤器、模式 | 先环回，再全接收，再查中断/FIFO |
| 偶发 CRC/Stuff Error | 波形质量、采样点、终端电阻 | 示波器看差分波形；测终端；调整位时序 |
| 一接入某节点整网异常 | 该节点波特率错误或硬件故障 | 单独断开节点，读 TEC/REC |
| 低优先级报文延迟大 | 总线负载高或 ID 规划不合理 | 统计总线负载，调整周期和 ID |
| Bus-Off | 持续发送错误 | 读错误寄存器，检查物理层和波特率 |

### 9.3 调试工具

- **示波器**：看 `CAN_H/CAN_L` 电平、差分幅值、边沿质量、反射。
- **CAN 分析仪**：看 ID、DLC、Data、周期、错误帧、总线负载。
- **逻辑分析仪**：可辅助看 TX/RX 逻辑侧，但不能替代差分波形检查。
- **万用表**：断电测终端电阻，上电测供电和共模电压。
- **固件日志**：记录发送失败、FIFO 溢出、TEC/REC、Bus-Off 次数。

## 十、速记表

### 10.1 核心概念

| 概念 | 速记 |
|:---|:---|
| 显性位 | 逻辑 0，强，任一节点发送 0 总线就是 0 |
| 隐性位 | 逻辑 1，弱，所有节点都释放才是 1 |
| ID | 消息含义 + 优先级，不是节点地址 |
| 仲裁 | ID 越小优先级越高，失败节点转接收 |
| ACK | 任意正确接收节点在 ACK 槽拉显性位 |
| 位填充 | 连续 5 个相同位后插入反向位 |
| Bus-Off | TEC >= 256，节点退出总线 |
| 终端电阻 | 高速 CAN 总线两端各 120Ω，等效约 60Ω |

### 10.2 STM32 初始化顺序

1. 使能 GPIO、AFIO、CAN 时钟。
2. 配置 `CAN_TX` 为复用推挽输出，`CAN_RX` 为输入。
3. 进入 CAN 初始化模式。
4. 配置工作模式、自动重传、自动恢复、发送优先级。
5. 配置位时序：`BRP`、`BS1`、`BS2`、`SJW`。
6. 配置过滤器和 FIFO 归属。
7. 使能接收中断、错误中断或发送中断。
8. 退出初始化模式，进入正常模式。
9. 发送前检查邮箱，接收时及时释放 FIFO。
10. 周期性监控错误状态和 Bus-Off。


- CAN 靠差分信号抗干扰。
- CAN 靠显性/隐性的线与特性实现非破坏性仲裁。
- CAN 靠 CRC、ACK、位填充、错误帧和错误计数保证可靠性。
- CAN 靠位时序和再同步解决异步通信采样问题。
