## 2. 数值类型转换

所有**标定参数**必须在*设计阶段手工计算*并写死为整数常量，不允许依赖编译器的浮点常量折叠行为。

> [!warning] 浮点 → 整数截断
> 浮点型变量转换为整数变量，小数部分会直接截断。
>
> **举例**：车窗电机正常运行时电流约 $0.5\text{A} = 500\text{mA}$。
> - 寄存器值 = $5000$（因为 $\text{LSB} = 100\mu\text{A}$，$500\text{mA} / 0.1\text{mA} = 5000$）
> - `(float)5000 * 0.0001f = 0.5f`
> - 赋值给 `uint16_t Current` → ==截断为 0！==

> [!tip] 最佳实践
> 整型参与浮点型计算时，应显式转换 `(float)`，做到见名知意。

## 3. 多个 `if` vs `if + else if`

### 核心区别

#### 多个独立的 `if`（无 `else`）

```c
if (条件A) { 动作A; }
if (条件B) { 动作B; }
if (条件C) { 动作C; }
```

- **每个条件都会被独立判断**，即使条件 A 已成立，B 和 C 仍会被检查。
- 多个条件同时成立时，**所有对应的代码块都会执行**。

#### `if + else if` 链

```c
if (条件A)      { 动作A; }
else if (条件B) { 动作B; }
else if (条件C) { 动作C; }
else            { 默认动作; }
```

- 一旦某个条件匹配，执行后**跳过整个链**，不再判断后续条件。
- **最多只有一个分支被执行**。

### 对比示例

假设 `x = 10, y = 5, z = 3`：

**多个 `if`** → 输出 `ABC`

```c
if (x > 5)  { printf("A"); }  // ✓
if (y > 2)  { printf("B"); }  // ✓
if (z > 1)  { printf("C"); }  // ✓
```

**`if-else if`** → 输出 `A`

```c
if (x > 5)      { printf("A"); }  // ✓ 命中，后续跳过
else if (y > 2) { printf("B"); }
else if (z > 1) { printf("C"); }
```

### 嵌入式场景选择

> [!example] 故障检测：用多个独立 `if`
> ```c
> // 每个故障都要独立检查，可能同时触发多个
> if (current > OVERCURRENT)  { Motor_Stop(); }
> if (limit_switch == PRESSED) { Motor_Stop(); }
> if (timeout_flag == SET)     { Motor_Stop(); }
> ```

> [!example] 模式选择：用 `if-else if`
> ```c
> // 一次只可能有一个命令
> if (cmd == CMD_AUTO_UP)       { AutoUp_Start(); }
> else if (cmd == CMD_AUTO_DOWN) { AutoDown_Start(); }
> else if (cmd == CMD_MANUAL)   { Manual_Control(); }
> else                           { InvalidCmd(); }
> ```

### 常见陷阱

> [!failure] 用多个 `if` 实现互斥
> ```c
> if (x > 5)   state = STATE_A;
> if (x <= 5)  state = STATE_B;  // 两个都执行！
> ```
> 应改为 `if-else`。

> [!failure] 用 `else if` 处理独立条件
> ```c
> if (current > OVERCURRENT) { Motor_Stop(); }
> else if (limit_switch)     { Motor_Stop(); }  // 过流时限位被跳过！
> ```

### 总结

| 需求 | 推荐 |
|:-----|------|
| 条件互斥，只选其一 | `if-else if` 或 `switch-case` |
| 条件独立，可能同时成立 | 多个独立 `if` |
| 有优先级，命中即停 | `if-else if`（高优先级在前） |

> [!info] 汽车嵌入式推荐
> 状态机状态切换、命令响应 → `if-else if`；故障检测、保护触发 → 多个独立 `if`。

## 4. 结构体 `.` 和 `->` 的区别

| 传参方式 | 访问成员 | 场景 |
|:---------|:---------|------|
| 结构体变量 `struct_t data` | `data.member` | 只读、不修改原值 |
| 结构体指针 `struct_t *p` | `p->member` | 需要在函数内修改原值 |

> [!tip] 记忆口诀
> 指针用箭头 `->`，变量用点 `.`。传来的是地址 → 改的就是原值。

## 5. 指针声明、类型转换与解引用

三种语法形似但含义完全不同：

| 语法 | 含义 | 示例 |
|:-----|:-----|------|
| `char *p` | **声明**指针变量 | `char *buf = &data;` |
| `(char *)p` | **强制类型转换**，把 `p` 转成 `char*` 类型 | `memcpy((char *)&reg, src, n);` |
| `*p` | **解引用**，取出指针指向的值 | `uint8_t val = *p;` |

> [!important]
> 对于指针而言，最不重要的是它本身的地址，**最重要**的是其**指向的地址和地址上的值**。

## 6. switch-case 穿透 (Fall-Through)

### 问题复现

在 [[自动车窗防夹模块]] 的 `AntiPinch_Process()` 中，`case AP_BLANKING:` 末尾缺少兜底返回，导致消隐期内执行流穿透到检测逻辑：

```c
case AP_BLANKING:
    if (is_rising == 0) { ... return 0; }   // 路径A：已停止
    if (时间到了)       { ... return 0; }   // 路径B：消隐结束
    // 路径C：is_rising==1 且时间未到
    // ↓ 无 return，无 break，直接穿透！

case AP_MONITORING:                          // ← CPU 走到了这里
    delta_ma = current - baseline;           // ← 开始检测防夹！
```

### 关键认知

`ap->AP_state` 的值仍是 `AP_BLINKING`（变量没变），但 **CPU 只看 switch 跳转表，不检查变量的值**。没有 `break` 或 `return`，就一直往下执行。

> [!bug] 实际后果
> 消隐期内电机启动产生尖峰电流 → 穿透到 `AP_MONITORING` → 阈值判断通过 → ==误触发防夹反转==。

### 修复方案

```c
case AP_BLANKING:
    if (is_rising == 0) { ... return 0; }
    if (时间到了)       { ... return 0; }
    return 0;  /* 消隐期进行中，本次不检测，等下次调用 */
```

> [!tip] 为什么不用 `break;`
> 编译器可能警告 `break` 后代码不可达。`return 0;` 明确表达"本次不检测"的意图，比 `break;` 语义更清晰。

### 检查清单

> [!todo] switch-case 自查清单
> - [ ] 每个 case 的**所有条件路径**都有 `return` 或 `break` 吗？
> - [ ] 有 `if-else` 分支时，`else` 路径有显式返回吗？
> - [ ] 多个 `if` 各自 `return` 了，最后一个 `if` 之后有兜底 `return` 吗？
> - [ ] 如果不能确定，先加兜底 `return`，再考虑优化掉。

> [!important] 通用原则
> **switch-case 中每个 case 结尾必须有 `break;` 或 `return;`。**
> 如果分支条件覆盖不全，最后一个条件之后加兜底 `return`，既防穿透又让意图明确。

## 7. 字符串类型选择

| 使用场景 | 声明方式 | 存储位置 | RAM 占用 |
|:---------|:---------|:---------|:---------|
| 只读常量（不修改） | `const char *p = "MAIN_UP";` | FLASH (.rodata) | 4 字节（指针） |
| 需要修改/拼接/接收数据 | `char buf[16];` | RAM（栈/全局区） | 连续 N 字节 |

### `const` 的三大意义

1. **编译保护**：误写会报错，防止破坏 FLASH
2. **存储优化**：放入 `.rodata`，节省宝贵 RAM（STM32F103 仅 20KB）
3. **代码自文档**：明确"不修改"，提高可读性

### 决策流程

```
需要修改字符串内容？
   ├── 是 → 必须用 char buf[N]（数组）
   └── 否 → 字符串来源？
               ├── 编译期已知常量 → const char *p（最佳）
               ├── 运行时生成/外部输入 → char buf[N]
               └── 返回局部字符串 → static char buf[N]
```

> [!tip] 一句话总结
> 不改内容用指针（省 RAM），需要写入用数组（占 RAM）。指针存地址，数组存内容。
