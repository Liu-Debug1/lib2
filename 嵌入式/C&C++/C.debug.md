## 2. 数值类型转换

所有**标定参数**必须在*设计阶段手工计算*并写死为整数常量，不允许依赖编译器的浮点常量折叠行为。

> [!warning] 浮点 → 整数截断
> 浮点型变量转换为整数变量，小数部分会直接截断。
>
> **举例**：车窗电机正常运行时电流约 $0.5\text{A} = 500\text{mA}$。
> - 寄存器值 = $5000$（因为 $\text{LSB} = 100\mu\text{A}$，$500\text{mA} / 0.1\text{mA} = 5000$）
> - `(float)5000 * 0.0001f = 0.5f`
> - 赋值给 `uint16_t Current` → ==截断为 0！==

> [!tip] 最佳实践
> 整型参与浮点型计算时，应显式转换 `(float)`，做到见名知意。

## 3. 分支选择：多个 `if` vs `if + else if` vs `switch-case`

### 核心区别

#### 多个独立的 `if`（无 `else`）

```c
if (条件A) { 动作A; }
if (条件B) { 动作B; }
if (条件C) { 动作C; }
```

- **每个条件都会被独立判断**，即使条件 A 已成立，B 和 C 仍会被检查。
- 多个条件同时成立时，**所有对应的代码块都会执行**。

#### `if + else if` 链

```c
if (条件A)      { 动作A; }
else if (条件B) { 动作B; }
else if (条件C) { 动作C; }
else            { 默认动作; }
```

- 一旦某个条件匹配，执行后**跳过整个链**，不再判断后续条件。
- **最多只有一个分支被执行**。

### 对比示例

假设 `x = 10, y = 5, z = 3`：

**多个 `if`** → 输出 `ABC`

```c
if (x > 5)  { printf("A"); }  // ✓
if (y > 2)  { printf("B"); }  // ✓
if (z > 1)  { printf("C"); }  // ✓
```

**`if-else if`** → 输出 `A`

```c
if (x > 5)      { printf("A"); }  // ✓ 命中，后续跳过
else if (y > 2) { printf("B"); }
else if (z > 1) { printf("C"); }
```

### 嵌入式场景选择

> [!example] 故障检测：用多个独立 `if`
> ```c
> // 每个故障都要独立检查，可能同时触发多个
> if (current > OVERCURRENT)  { Motor_Stop(); }
> if (limit_switch == PRESSED) { Motor_Stop(); }
> if (timeout_flag == SET)     { Motor_Stop(); }
> ```

> [!example] 模式选择：用 `if-else if`
> ```c
> // 一次只可能有一个命令
> if (cmd == CMD_AUTO_UP)       { AutoUp_Start(); }
> else if (cmd == CMD_AUTO_DOWN) { AutoDown_Start(); }
> else if (cmd == CMD_MANUAL)   { Manual_Control(); }
> else                           { InvalidCmd(); }
> ```

### 常见陷阱

> [!failure] 用多个 `if` 实现互斥
> ```c
> if (x > 5)   state = STATE_A;
> if (x <= 5)  state = STATE_B;  // 两个都执行！
> ```
> 应改为 `if-else`。

> [!failure] 用 `else if` 处理独立条件
> ```c
> if (current > OVERCURRENT) { Motor_Stop(); }
> else if (limit_switch)     { Motor_Stop(); }  // 过流时限位被跳过！
> ```

### if-else vs switch-case 选择

| 场景 | 推荐 |
|:-----|:-----|
| 有限个固定状态/值 | `switch-case` + 枚举 |
| 范围判断、复杂条件 | `if-else if` |

> [!tip] 状态机最佳实践
> 1. 建立枚举 → 列出所有状态
> 2. `switch-case` → 执行每个状态下的任务
> 3. 状态机判断**不要放在中断里**执行，程序太长会阻塞其他中断

### 总结

| 需求 | 推荐 |
|:-----|------|
| 条件互斥，只选其一 | `if-else if` 或 `switch-case` |
| 条件独立，可能同时成立 | 多个独立 `if` |
| 有优先级，命中即停 | `if-else if`（高优先级在前） |

> [!info] 汽车嵌入式推荐
> 状态机状态切换、命令响应 → `if-else if`；故障检测、保护触发 → 多个独立 `if`。

## 4. 结构体 `.` 和 `->` 的区别

| 传参方式 | 访问成员 | 场景 |
|:---------|:---------|------|
| 结构体变量 `struct_t data` | `data.member` | 只读、不修改原值 |
| 结构体指针 `struct_t *p` | `p->member` | 需要在函数内修改原值 |

> [!tip] 记忆口诀
> 指针用箭头 `->`，变量用点 `.`。传来的是地址 → 改的就是原值。

## 5. 指针声明、类型转换与解引用

三种语法形似但含义完全不同：

| 语法 | 含义 | 示例 |
|:-----|:-----|------|
| `char *p` | **声明**指针变量 | `char *buf = &data;` |
| `(char *)p` | **强制类型转换**，把 `p` 转成 `char*` 类型 | `memcpy((char *)&reg, src, n);` |
| `*p` | **解引用**，取出指针指向的值 | `uint8_t val = *p;` |

> [!important]
> 对于指针而言，最不重要的是它本身的地址，**最重要**的是其**指向的地址和地址上的值**。

### 指针操作四步

| 语法 | 含义 | 示例 |
|:-----|:-----|:-----|
| `&变量` | 获取变量的地址 | `p = &a;` |
| `指针` | 存储了地址的变量 | `int *p;` |
| `*指针` | 访问指针指向的值 | `val = *p;` |
| `&指针` | 获取指针自己的地址（极少用） | `int **pp = &p;` |

```c
// 指针操作完整示例
uint8_t a = 10;
uint8_t *p = &a;    // p 存储 a 的地址
*p = 20;            // 通过指针修改 a → a 变成 20
uint8_t b = *p;     // 通过指针读取 → b = 20
```

### 空指针保护

```c
uint8_t *p = NULL;  // NULL = 0x00000000，STM32 中地址 0 不可访问

if (p == NULL) return 0;  // 先判断，再解引用
```

> [!danger] 不保护空指针的后果
> 如果直接 `*p = 1` 写入 0x00000000：
> 1. 可能破坏向量表
> 2. 程序跑飞，行为不可预测
> 3. 最坏情况：需要重新烧录程序

## 6. switch-case 穿透 (Fall-Through)

### 问题复现

在 [[自动车窗防夹模块]] 的 `AntiPinch_Process()` 中，`case AP_BLANKING:` 末尾缺少兜底返回，导致消隐期内执行流穿透到检测逻辑：

```c
case AP_BLANKING:
    if (is_rising == 0) { ... return 0; }   // 路径A：已停止
    if (时间到了)       { ... return 0; }   // 路径B：消隐结束
    // 路径C：is_rising==1 且时间未到
    // ↓ 无 return，无 break，直接穿透！

case AP_MONITORING:                          // ← CPU 走到了这里
    delta_ma = current - baseline;           // ← 开始检测防夹！
```

### 关键认知

`ap->AP_state` 的值仍是 `AP_BLINKING`（变量没变），但 **CPU 只看 switch 跳转表，不检查变量的值**。没有 `break` 或 `return`，就一直往下执行。

> [!bug] 实际后果
> 消隐期内电机启动产生尖峰电流 → 穿透到 `AP_MONITORING` → 阈值判断通过 → ==误触发防夹反转==。

### 修复方案

```c
case AP_BLANKING:
    if (is_rising == 0) { ... return 0; }
    if (时间到了)       { ... return 0; }
    return 0;  /* 消隐期进行中，本次不检测，等下次调用 */
```

> [!tip] 为什么不用 `break;`
> 编译器可能警告 `break` 后代码不可达。`return 0;` 明确表达"本次不检测"的意图，比 `break;` 语义更清晰。

### 检查清单

> [!todo] switch-case 自查清单
> - [ ] 每个 case 的**所有条件路径**都有 `return` 或 `break` 吗？
> - [ ] 有 `if-else` 分支时，`else` 路径有显式返回吗？
> - [ ] 多个 `if` 各自 `return` 了，最后一个 `if` 之后有兜底 `return` 吗？
> - [ ] 如果不能确定，先加兜底 `return`，再考虑优化掉。

> [!important] 通用原则
> **switch-case 中每个 case 结尾必须有 `break;` 或 `return;`。**
> 如果分支条件覆盖不全，最后一个条件之后加兜底 `return`，既防穿透又让意图明确。
>
> **状态机代码不要放在中断服务函数里执行**——ISR 必须快进快出，长逻辑放主循环轮询。

## 7. 字符串类型选择

| 使用场景 | 声明方式 | 存储位置 | RAM 占用 |
|:---------|:---------|:---------|:---------|
| 只读常量（不修改） | `const char *p = "MAIN_UP";` | FLASH (.rodata) | 4 字节（指针） |
| 需要修改/拼接/接收数据 | `char buf[16];` | RAM（栈/全局区） | 连续 N 字节 |

### `const` 的三大意义

1. **编译保护**：误写会报错，防止破坏 FLASH
2. **存储优化**：放入 `.rodata`，节省宝贵 RAM（STM32F103 仅 20KB）
3. **代码自文档**：明确"不修改"，提高可读性

### 决策流程

```
需要修改字符串内容？
   ├── 是 → 必须用 char buf[N]（数组）
   └── 否 → 字符串来源？
               ├── 编译期已知常量 → const char *p（最佳）
               ├── 运行时生成/外部输入 → char buf[N]
               └── 返回局部字符串 → static char buf[N]
```

> [!tip] 一句话总结
> 不改内容用指针（省 RAM），需要写入用数组（占 RAM）。指针存地址，数组存内容。

## 8. 局部变量与 static

### 8.1 局部变量必须手动初始化

函数内定义的局部变量存放在栈上，C 语言不会自动清零，**必须手动初始化**，否则值是随机的。

```c
// ❌ 错误：未初始化，值是随机的
void func(void) {
    uint8_t count;
    count++;  // 垃圾值 + 1，行为不可预测
}
// ✅ 正确：显式初始化
void func(void) {
    uint8_t count = 0;
    count++;
}
```

### 8.2 static 变量放哪里？

| 使用场景 | 定义位置 |
|:---------|:---------|
| 仅一个函数用 | 函数内部（局部静态） |
| 多个函数共用 | 文件顶部（静态全局） |

> 函数内的变量，外部永远碰不到。

### 8.3 四种情况完整对比

| | 不加 static | 加 static |
|:--|:------------|:----------|
| **函数内变量** | 栈内存，函数返回销毁，每次重新初始化 | 静态存储区，程序运行期间保留，仅初始化 1 次 |
| **文件顶部变量** | 外部链接，其他文件可 `extern` 访问 | **内部链接**，仅本文件可见 |
| **函数** | 外部链接，声明后可跨文件调用 | 内部链接，其他文件不可调用 |

> 函数内静态变量 = **只初始化 1 次 + 值一直保留 + 作用域仅函数内**。

### 8.4 全局命名空间污染

不加 `static` 的函数/变量会进入**全局符号表**，带来三大问题：

| 问题 | 后果 |
|:-----|:-----|
| 命名冲突 | 多个文件同名函数 → 链接器不知道执行哪一个 |
| 破坏封装 | 其他模块也能看到/调用内部函数，增加耦合 |
| 增加维护风险 | 修改一个内部函数可能引发多模块报错 |

> [!tip] 习惯
> 编写 `.c` 文件时，不打算通过 `.h` 公开的函数和变量，**第一时间加 `static`**。这是工业界硬性要求。

## 10. volatile 关键字

| 作用 | 说明 |
|:-----|:-----|
| 直接从内存访问 | 跳过 CPU 缓存/寄存器优化，每次读取都走总线 |
| 防止编译器优化 | 保证实时性，不会把变量优化成常量 |
| 主要用于 | 中断标志位、主循环与 ISR 共享的变量 |
| 保证 | 中断和主循环数据同步（可见性） |

```c
volatile uint8_t flag = 0;  // ISR 写，主循环读

// 不加 volatile 时，编译器可能把 while(flag==0) 优化成死循环
while (flag == 0);  // 等待中断置位
```

## 11. 串口字符串处理

### 11.1 回显调试

串口不通时，在接收中断里加回显函数 `Serial_SendByte(RxData);`，确认单片机是否收到数据。

### 11.2 换行符兼容

Windows 换行符是 `\r\n`，但不同设备可能有 `\r`、`\n`、`\r\n` 三种情况。状态机判断包尾时，**碰到 `\r` 或 `\n` 其中一个就认为包尾结束**。

### 11.3 strcmp vs strstr

| 函数 | 匹配方式 | 适用场景 |
|:-----|:---------|:---------|
| `strcmp(buf, "LED_ON")` | 严格全等 | 数据包格式固定 |
| `strstr(buf, "LED_ON")` | 包含即匹配 | 数据包内容可变 |

### 11.4 strstr 包含关系陷阱

当 `"MAIN_UP"` 和 `"MAIN_UP_HOLD"` 同时存在，`strstr` 会误判（"MAIN_UP_HOLD" 中包含 "MAIN_UP"）。

| 解决方法 | 做法 |
|:---------|:-----|
| **调整判断顺序** | 长字符串放在短字符串之前（先比 "MAIN_UP_HOLD"，再比 "MAIN_UP"） |
| **更换名称** | 断绝包含关系，如 `MAIN_HOLD` → `MAIN_HOLD` |

## 12. 前后台系统架构

| | 定义 | 示例 |
|:--|:-----|:-----|
| **前台** | 中断服务函数（ISR），快进快出 | `TIM2_IRQHandler`、`EXTI_IRQHandler`、串口中断 |
| **后台** | `main()` 的 `while(1)` 轮询 | 各任务模块按顺序执行 |

### 六大准则

1. **执行顺序**：采样输入 → 计算逻辑 → 限位保护 → 输出执行
2. **模块化封装**：每个功能模块一个 `.c/.h`，主程序只调用
3. **合理使用 static**：内部函数和变量加 `static`，不污染全局命名空间
4. **主循环全部非阻塞**：不允许 `delay_ms()` 或死等，用时间戳驱动
5. **共享变量加 volatile**：中断和主循环共用的标志位必须加
6. **加看门狗**：做产品必须加独立/窗口看门狗，防电磁干扰导致程序卡死

### 主程序代码结构

```c
// main.c — 推荐的结构顺序

/* ===== 1. 宏定义 ===== */
#define LED_PIN GPIO_Pin_13

/* ===== 2. 外部变量声明 ===== */
extern volatile uint32_t g_SysTick;

/* ===== 3. 全局变量定义 ===== */
static uint8_t g_Flag = 0;

/* ===== 4. 函数声明 ===== */
static void System_Init(void);
static void Task_LED(void);

/* ===== 5. main() ===== */
int main(void) {
    System_Init();
    while (1) {
        Task_LED();
    }
}

/* ===== 6. 函数实现 ===== */
static void System_Init(void) { /* ... */ }
static void Task_LED(void) { /* ... */ }
```

> 顺序：**宏定义 → 外部变量声明 → 全局变量定义 → 函数声明 → main() → 函数实现**。
