## 一、变量命名规范

使用**三段式大驼峰**命名：`(1)_(2)_(3)`

- **(1) 为变量的作用范围**：`gXXX`（global）在这个终端所在的工程包含的所有 `.c` 均可使用，XXX 为该变量所在模块，如 [[#表1：模块简写说明|表1]] 所示；`l`（local）为仅在函数内部可使用的局部变量，`m`（member）为类中可使用的变量。
- **(2) 为变量的具体含义**：使用首字母大写的大驼峰命名规则，表示变量的具体含义。为确保变量名尽可能短，建议使用一定的缩写，例如 `VehPosF`（VehiclePositionFiltered）、`SteerAngCmdFf`（SteerAngleCommandFeedforward）。
- **(3) 为变量的声明类型**：例如 `b` 为 bool，`u8` / `u16` / `u32` 表示无符号 8 位 / 16 位 / 32 位，`s8` / `s16` / `s32` 表示有符号 8 位 / 16 位 / 32 位，`db` 表示 double；指针的表示方式为 `p` + 所指向的数据类型，如 `ps8`，表示 s8 数组的指针。

**类型后缀速查：**

| 后缀 | 含义 |
|------|------|
| `b` | bool |
| `u8` / `u16` / `u32` | 无符号 8/16/32 位 |
| `s8` / `s16` / `s32` | 有符号 8/16/32 位 |
| `db` | double |
| `p` + 类型 | 指针，如 `ps8` 表示 s8 数组指针 |

**示例：**

```c
gLC_VehPosF_db    // 全局-定位-车辆位置滤波(double)
```

## 二、函数命名规范

使用**两段式小驼峰**命名：`(1)_(2)`

- **(1) 为函数的作用范围**：XXX 表示在这个终端所在的工程包含的所有 `.c` 均可使用，参见 [[#表1：模块简写说明|表1]]；`m`（member）为类中可使用的函数。
- **(2) 为函数的具体含义**：需要遵循以下要求：

**函数名通常以一个强动词开头**，以清晰地表达其执行的动作：

- **查询/获取类：**
  - `get...()` — 获取一个值
  - `fetch...()` — 从外部源获取（可能耗时）
  - `query...()` — 执行查询（通常返回集合）
  - `find...()` — 查找特定元素
  - `calculate...()` / `compute...()` — 计算一个值
  - `is...()` / `has...()` / `can...()` — 返回布尔值，表示状态或能力
    - `isEmpty()`
    - `hasAdminPrivileges()`
    - `canBeDeleted()`
- **修改/设置类：**
  - `set...()` — 设置一个属性
  - `update...()` — 更新一个对象的状态
  - `apply...()` — 应用一个变化或配置
  - `add...()` / `insert...()` — 增加一个元素
  - `remove...()` / `delete...()` / `erase...()` — 移除一个元素
- **执行/操作类：**
  - `open...()` / `close...()`
  - `start...()` / `stop...()`
  - `process...()`
  - `initialize...()` / `shutdown...()`

**具体的命名要求：**

1. 遵循小驼峰法，即将每个单词的首字母小写，并将单词连在一起，单词之间不使用下划线。示例：`getUserById()`、`processPayment()`、`setBackgroundColor()`、`isFileExist()`。
2. 使用动词开头：
   - `get...()` — 获取一个值
   - `find...()` — 查找特定元素
   - `calc...()` — 计算
   - `set...()` — 设置一个属性
   - `update...()` — 更新一个对象的状态
   - `is...()` — 判断是否成立
   - `init...()` — 初始化

> [!NOTE]
> 单词之间不使用下划线，直接连写，首字母小写。

## 三、类的命名和定义要求

1. 类的声明采用**单段式声明、大驼峰命名**方式，如 `UserProfile`、`PaymentService`。
2. 实际可执行的类与变量的定义方式类似，采用**两段式大驼峰** `(2)_(3)`，式中 (2) 为类名，(3) 表示变量类型为类，固定采用 `C`，如 `RollerPlanner_C`。
3. 类中的函数严格参照函数命名要求，例如初始化变量均放在以 `init` 开头的函数中。

## 四、工程的要求

**工程的定义**：指在一个终端内可单独运行的多个 `.c` 和 `.h` 文件组成，工程的所有文件需放在同一个文件夹下，每个工程只允许有一个 `main` 函数。

**交互要求**：工程与工程之间的变量交互只允许通过消息。

**时序控制**：由于每个工程都由一个终端单独控制，因此工程与工程间的时序无法严格保证，如果需要控制多个功能块之间的时序，则多个功能块必须放在同一个工程下。

**main.c 的结构**：`main` 函数通常只包括初始化、时序控制两部分，其余代码不放在 `main.c` 文件中。

## 五、注释规范

### 基本原则

**1. 注释是为了解释"为什么"，而不是"是什么"。**

- 代码本身应该足够清晰，能解释"是什么"和"怎么做"。好的变量名、函数名和类名本身就是最好的注释。
- 注释应该解释那些从代码本身看不出的业务背景、设计决策、历史原因或潜在陷阱。
- 反例：`// i = i + 1; // 将i加1`（这是废话）
- 正例：`// 重试计数器，最多允许3次失败后放弃`（解释了为什么是3）

**2. 保持注释的简洁和准确。**

- 避免冗长、模糊的描述。注释应该像代码一样精炼。
- 确保注释与代码保持同步。过时的、错误的注释比没有注释更糟糕。

**3. 优先让代码自解释。**

- 在写注释之前，先思考是否可以通过重构代码（例如提取函数、使用更具描述性的变量名）来让代码本身更清晰。

- 关于 BUG 的修改，直接在修改上标注 BUG 编号、并在 Git push 时也标注 BUG 编号。
- 变量的注释使用单行注释 `//`
- 函数的注释使用多行注释，并放在函数声明的开头，例如：

```c
/*
功能: 比例阀的PID控制
定义: WLT, 2020.06.11
修改: WLT, 2020.06.11
*/
```

- 函数内具体的注释，使用 `// #1.1` 的方式，来表示具体的代码结构。

**示例：**


## 六、宏定义和头文件的使用

### 建议使用宏定义的场合

- 模式切换
- 大量被重复调用的同一个参数

### 命名规则

使用**全大写 + 下划线分割**，如 `MAX_BUFFER_SIZE`。

### 模式切换示例

用于模式切换的宏定义，请对模式进行预先定义，例如：

```c
#define PLT_VEHICLE    0  // 实车测试
#define PLT_SIMULATION 1  // 仿真
#define PLT_MODE       PLT_VEHICLE
```

> [!NOTE] 数学函数也可以用宏定义
> 一些常用的数学函数，也可以使用宏定义。

**常用宏定义示例：**

```c
#ifndef MIN
#define MIN(a, b)  ( ((a) < (b)) ? (a) : (b) )
#endif

#ifndef MAX
#define MAX(a, b)  ( ((a) > (b)) ? (a) : (b) )
#endif

#ifndef A_MAX_AB
#define A_MAX_AB(a, b)  { if ((a) < (b)) { (a) = (b); } }
#endif

#ifndef A_MIN_AB
#define A_MIN_AB(a, b)  { if ((a) > (b)) { (a) = (b); } }
#endif

#ifndef A_MAX_BC
#define A_MAX_BC(a, b, c)  { (a) = (b); if ((c) > (b)) { (a) = (c); } }  // bosch definition
#endif

#ifndef A_MIN_BC
#define A_MIN_BC(a, b, c)  { (a) = (b); if ((c) < (b)) { (a) = (c); } }  // bosch definition
#endif

#ifndef BOUND
#define BOUND(a, MAX, min)  { if ((a) > (MAX)) { (a) = (MAX); } if ((a) < (min)) { (a) = (min); } }
#endif
```

### h 文件主要包含内容

- 大量相同的 `.h` 包含
- 宏定义
- 函数的声明

> [!WARNING] 宏定义只在一处定义
> 由于宏定义的使用范围较广，通常在 `.h` 文件中定义，并且只在一个 `.h` 文件中定义。需要使用该宏定义时 `include` 该 `.h` 文件即可。

## 七、代码的主体结构

**核心原则：**

采用顺序结构，即**初始化 → 主循环 → 后处理**。

**代码组织原则：**

1. **自上而下，分层组织**：将文件从"高层概念"到"底层实现"进行组织。读者应该先看到接口和摘要，再看到细节。
2. **逻辑分组**：将相关的代码（如包含的头文件、成员函数、私有数据）放在一起，并用注释块明确分隔。
3. **关注点分离**：将不同职责的代码分离开。例如，公共接口、私有实现、辅助函数应该有明确的界限。
4. **一致性**：在整个项目中，所有源文件的结构必须保持一致。

主循环内部同样采用顺序结构，如必要采用分支结构时，需从同一个起点开始，并汇总到同一个终点，随后继续采用顺序结构。

## 八、其他

> [!NOTE] 以下内容待进一步完善

- **录数据**：统一在一个文件录 `data_log`
- **规定好架构**：文件名、目录、各工程功能，只在架构里面修改相应代码
- **头文件统一**：头文件放在统一一个头文件工程里，提高可移植性

**全程序时间统一：**

设置一个单独函数算时间，全局代码都调用这个时间，正常程序是主函数不复制不判断直接调用函数。

1. 再搞个时钟，不用到 ROS 上取时钟。时钟在主程序里算，程序如果一次运行是 0.01s，时钟每调用一次 +0.01s。
2. 主函数的调用频率原先是 10Hz，改为 100 倍（宏定义定义仿真的倍率），算时钟的时候调用原先的时钟，然后通过主函数获取乘以 100 倍 dt。
3. 调用的时钟都是用第 2 步的时钟，不要用 ROS 的时钟。

**include 整理：**

把所有比较基础的库文件放在一个 `.h` 里，这样其他 `.c` 只需要包括这个共性的 `.h`，再单独包括几个独特的，就行了。

**文件组织：**

- 消息类型单独放在消息的 cpp 里面
- 纯公式计算放在计算的 cpp 里面.
